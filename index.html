1 · Hardware Profile
VRAM / Unified Mem (GB) i Total high-speed memory on your GPU(s) or Apple Silicon. The full model must fit here; spilling to system RAM causes a ~95% speed penalty on dense models (less severe with llama.cpp CPU offloading).
GPU Mem Bandwidth (GB/s) i How fast your GPU reads from VRAM. LLM decode is strictly memory-bandwidth-bound — every parameter must be read per token generated. Examples: RTX 4090 = 1008 GB/s · A100 80GB = 2000 GB/s · M3 Max = 300 GB/s.
Enable CPU Offload (llama.cpp / Ollama style) i When the model exceeds GPU VRAM, inference engines like llama.cpp split layers between GPU and CPU. Speed becomes a weighted harmonic mean of both bandwidths. Enables running large models on modest GPUs.
CPU Mem Bandwidth (GB/s) iSystem RAM bandwidth. DDR5 is ~80 GB/s, DDR4 ~50 GB/s. This is the bottleneck for CPU-side layers. Apple M-series unified memory is already accounted for in the GPU bandwidth field.
System RAM (GB) iTotal available system RAM for CPU-offloaded layers and overflow. Only counts toward feasibility if CPU offload is enabled.

2 · Model Target
Architecture iDense: every parameter is used per token. MoE (Mixture of Experts): only a subset of "expert" layers activate per token, dramatically cutting bandwidth cost at inference while keeping total parameter count high.
Quantization (Weights) iBit-precision for model weights. Lower = smaller VRAM footprint and faster token generation, with minor quality trade-offs. Q4 is the community sweet spot. FP16 is the lossless baseline.
Total Parameters (Billions) iTotal model weight count. For dense models, ALL of these are active every token. E.g. Llama 3.1 70B = 70.
⚡ MoE Mode Only active parameters consume bandwidth per token
Total Parameters (B) iAll parameters that must be loaded into memory. E.g. Gemma 4 "26B" has 26B total weights. Qwen3 80B has 80B total weights. All must fit in memory, but only active params use bandwidth.
Active Parameters (B) — "AXB" iParameters activated per token decode step. Shown as "AXB" in model names: Gemma 4 26B A4B → 4B active. Qwen3 80B A22B → 22B active. THIS drives your tokens/sec, not total params.
MoE sparsity ratio: active per token — bandwidth cost is of a comparably-sized dense model
3 · KV Cache
Context Window (Tokens) iConversation history length. KV cache memory grows linearly with context. Long contexts (>32k) can consume several GB even with quantization.
KV Cache Quantization illama.cpp supports quantizing the KV cache separately from weights (--cache-type-k / --cache-type-v flags). 4-bit KV can cut cache memory by 4× vs FP16 with minimal quality loss — critical for long contexts.
